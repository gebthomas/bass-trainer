# Not Now

This document should be read alongside `future_directions.md`.

An item appearing in `future_directions.md` is not necessarily in active scope.

This document captures ideas that may eventually become valuable but are intentionally deferred.

The purpose is not to reject these ideas permanently.

The purpose is to protect the project’s focus during the current phase:
- validating real musical usefulness
- improving practice workflows
- strengthening internalization and rehearsal readiness
- refining timing, groove, memory, and adaptive practice systems

An idea being listed here means:
> “interesting, but not currently the highest-leverage direction.”

---

# Current Priorities

The current priorities are:

- validating live musical usefulness
- improving practice continuity
- adaptive practice guidance
- groove and timing stability
- memory/internalization workflows
- repertoire maintenance
- ear training
- rehearsal readiness
- low-friction practice UX
- resilient musicianship under real ensemble conditions

Features that do not clearly strengthen these goals should generally wait.

---

# Deferred Technical Areas

## Heavy Machine Learning Systems

Examples:
- AI-generated improvisation critique
- deep performance scoring
- ML-based “musicality” evaluation
- large recommendation models

Reason deferred:
Simple and interpretable systems should be validated first.
The project should avoid unnecessary opacity and complexity before core workflows are proven useful.

---

## Advanced DSP Pipelines

Examples:
- polyphonic transcription
- sophisticated source separation
- studio-grade onset pipelines
- advanced spectral analysis
- high-end audio mastering features

Reason deferred:
Current goals are practice and musicianship support, not professional audio engineering.

Simple, robust approaches are preferred until limitations are clearly demonstrated.

---

## Full Automatic Transcription

Examples:
- complete bass transcription from recordings
- automatic notation generation
- DAW-like editing workflows

Reason deferred:
Useful eventually, but outside the current core focus.

The project is currently centered on:
- practice
- internalization
- rehearsal readiness
- adaptive coaching

not music production tooling.

---

## DAW Replacement Features

Examples:
- multitrack editing
- recording studio workflows
- plugin hosting
- mixing/mastering environments

Reason deferred:
This project is not intended to become a general music production platform.

---

## Social / Competitive Features

Examples:
- leaderboards
- public scoring
- competitive ranking
- social feeds
- practice “gamification”
- achievement systems

Reason deferred:
The project is intended to support long-term musicianship and productive practice, not competition or addictive engagement loops.

Positive reinforcement is valuable.
Competitive engagement mechanics are not a current priority.

---

## Cloud-First Architecture

Examples:
- mandatory online accounts
- cloud-dependent storage
- server-side analysis
- web-first infrastructure

Reason deferred:
The system should remain usable:
- locally
- offline
- with low setup friction

Cloud functionality may become useful later, but should not dominate early architecture decisions.

---

## Mobile-First Development

Examples:
- iOS-first workflows
- Android-first UX
- mobile optimization as primary target

Reason deferred:
Current experimentation and iteration are easier on desktop/laptop systems with audio interfaces and development tooling.

Mobile support may become important later.

---

## Large GUI Framework Investments

Examples:
- polished cross-platform UI systems
- advanced animation frameworks
- large frontend architecture work

Reason deferred:
The project is still validating core workflows and musical usefulness.

Architecture should remain lightweight until the core interaction model stabilizes.

---

## Plugin Ecosystems and Extensibility Frameworks

Examples:
- third-party plugin APIs
- scripting ecosystems
- extension marketplaces

Reason deferred:
Premature extensibility often increases complexity before the core product direction is mature.

The core workflows should stabilize first.

---

## Multiplayer / Realtime Collaboration

Examples:
- online rehearsals
- synchronized ensemble sessions
- shared practice rooms
- networked timing analysis

Reason deferred:
Interesting, but substantially outside the current scope.

The current focus is individual musicianship development and rehearsal preparation.

---

# Features Requiring Strong Justification

The following categories should require unusually strong justification before implementation:

- large persistent infrastructure
- cloud dependency
- opaque ML systems
- high-maintenance GUI frameworks
- features optimized for engagement rather than musicianship
- features that increase cognitive load during practice
- features that encourage dependency rather than internalization

---

# Reminder

The project’s primary value is not:
- technical novelty
- maximal feature count
- architectural complexity

The project’s value is:
- helping musicians practice more effectively
- strengthening internal pulse and groove
- improving memory and harmonic fluency
- supporting long-term repertoire maintenance
- improving adaptability in real ensemble settings
- helping musicians become more self-sufficient over time

When in doubt:
prefer simpler systems that improve real musical practice.
